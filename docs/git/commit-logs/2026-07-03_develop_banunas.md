## 4. 변경 배경 및 결과

기존에는 백엔드 플랜 문서와 동시성/레이스 컨디션 검토 문서가 별도로 존재해 최종 아키텍처가 한곳에 정리되어 있지 않았고, 팀 내 커밋 메시지 작성 방식도 정해진 규칙이 없어 일관성이 없었다.

이를 해결하기 위해 여러 대안 중 "Redis 원자 연산" 방식을 최종안으로 확정하고, 전체 시스템 구조(배포 토폴로지, 채널 레이어, 판정 로직 등)를 하나의 설계 문서(architecture.md)로 통합 정리했다. 또한 커밋 메시지 형식과 커밋 로그 작성 형식을 명문화한 문서(commit_FORMAT.md)를 추가했다.

결과적으로 팀원 모두가 참조할 수 있는 단일 아키텍처 기준 문서가 생겼고, 이후 커밋들은 정해진 형식을 따르게 되어 히스토리 추적과 협업이 쉬워졌다.

---

## 5. 중요 변경 사항

- 레이스 컨디션 대응 방식을 "대안 2: Redis 원자 연산"으로 최종 확정
- 배포 토폴로지를 VM(AWS EC2) + Cloudflare 프록시 구조로 정리 (기존 백엔드 플랜의 Railway 단일 배포안과 차이가 있음을 문서에 명시)
- 커밋 메시지 규칙(type/scope/subject/body/footer)과 커밋 로그 파일 구조(7개 섹션)를 신규 정의
