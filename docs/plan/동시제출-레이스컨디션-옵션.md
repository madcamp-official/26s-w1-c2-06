# 동시 제출(뺏어치기) 레이스 컨디션 방지 옵션

뺏어치기 판정(여러 클라이언트가 거의 동시에 같은 코드를 제출할 때 최초 제출자만 득점)을 어떻게 구현할지 검토한 문서. [architecture.md](./architecture.md) §7에서 이어지는 결론의 근거.

## 핵심 원칙

Django Channels(ASGI)는 방 하나에 여러 워커/프로세스가 붙을 수 있어 인메모리 변수로는 동시성을 보장할 수 없다. **"제출 타임스탬프 비교"가 아니라 "누가 먼저 원자적 연산에 성공했는가"로 판정한다** — 클라이언트/서버 타임스탬프는 네트워크 지터로 어긋날 수 있지만, Redis는 단일 스레드로 명령을 순차 실행하므로 "어떤 요청이 먼저 성공했는가"는 항상 명확하다.

## 검토한 옵션

1. **`SET NX` 단순 선점** — 구현 가장 단순하지만, 선점과 점수 반영이 Redis 왕복 2번으로 나뉘어 그 사이 서버가 죽으면 불일치 가능
2. **Lua Script (`EVAL`) — 채택** — 선점 확인 + 점수 반영을 하나의 원자 연산으로 묶어 중간 상태 불일치 없음. 왕복 횟수도 적어 지연시간에 유리
3. **Sorted Set + 타임스탬프 기록** — 모든 시도가 남아 "몇 ms 차이로 졌는지" 연출엔 유리하지만 구조가 가장 복잡. 이런 연출이 필요해지면 그때 추가 고려
4. **In-memory per-room lock** — Redis 왕복이 없어 빠르지만, 방을 특정 워커에 고정(sticky routing)해야 해서 Channels+Redis 채널 레이어의 분산 구조와 전제가 어긋남
5. **DB 레벨 락(`select_for_update`)** — 트랜잭션 오버헤드가 Redis보다 크고, 이미 Redis를 쓰는 상황에서 판정만 DB로 보낼 이유 없음 — 비추천

## 결론

**옵션 2(Lua Script)를 채택.** 선점+채점을 원자적으로 묶고, Channels가 이미 Redis를 channel layer로 쓰고 있어 추가 인프라 없이 적용 가능하기 때문. 실제 스크립트는 [backend-implementation.md](./backend-implementation.md) §5 참고.
